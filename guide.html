💡 Why Kubernetes?

          
Kubernetes automatically scales your containerized applications, provides self-healing, performs rolling updates, and works across multi-cloud environments. Instead of a single docker run, you manage your infrastructure as code through declarative YAML files.

        

      


      

        
📄 Step 1: Creating a Dockerfile

        
Everything starts with a Dockerfile. This file defines how your application is containerized.

        
Simple Node.js Application


# 1. Choose base image
FROM node:20-alpine
WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production
COPY . .
EXPOSE 3000
CMD ["node", "server.js"]

        
Multi-Stage Build (Production)


FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

FROM node:20-alpine
WORKDIR /app
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/package*.json ./
RUN npm ci --only=production
EXPOSE 3000
USER node
CMD ["node", "dist/server.js"]

        

          
✅ Best Practice

          
Use multi-stage builds to keep your final image small. Run as a non-root user with USER node. Use .dockerignore to exclude unnecessary files.

        

      


      

        
🐳 Step 2: Docker Image Build & Push


docker build -t myapp:1.0.0 .
docker run -p 3000:3000 myapp:1.0.0
docker tag myapp:1.0.0 ghcr.io/username/myapp:1.0.0
docker push ghcr.io/username/myapp:1.0.0

        
Popular container registries:

        
    
          
  • Docker Hub — docker.io/username/myapp
    • 
          
  • GitHub Container Registry — ghcr.io/username/myapp
    • 
          
  • AWS ECR — 123456789.dkr.ecr.region.amazonaws.com/myapp
    • 
          
  • Azure ACR — myregistry.azurecr.io/myapp
    • 
          
  • GCP Artifact Registry — region-docker.pkg.dev/project/repo/myapp
    • 
        

        

          
⚠️ Warning

          
Avoid the :latest tag! Always use semantic versioning (1.0.0). This is critical for Kubernetes rollback and reproducibility.

        

      


      
☸️ Step 3: Kubernetes Deployment


apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
  namespace: production
spec:
  replicas: 3
  selector:
    matchLabels:
      app: myapp
  template:
    metadata:
      labels:
        app: myapp
    spec:
      containers:
        - name: myapp
          image: ghcr.io/username/myapp:1.0.0
          ports:
            - containerPort: 3000
          resources:
            requests:
              cpu: "250m"
              memory: "256Mi"
            limits:
              cpu: "500m"
              memory: "512Mi"


      
🌐 Step 4: Service


apiVersion: v1
kind: Service
metadata:
  name: myapp-service
spec:
  type: ClusterIP
  selector:
    app: myapp
  ports:
    - port: 80
      targetPort: 3000

        
  • ClusterIP — Internal cluster access (default)
  • NodePort — Opens port on every node (30000-32767)
  • LoadBalancer — Gets a cloud load balancer

📏 Step 6: Resource Requests & Limits

        
This is the most critical part for cost estimation. Cloud providers charge based on requests:


resources:
  requests:          # Minimum guaranteed resources
    cpu: "250m"       # 0.25 vCPU
    memory: "256Mi"   # 256 MiB RAM
  limits:            # Maximum allowed resources
    cpu: "500m"       # 0.5 vCPU
    memory: "512Mi"   # 512 MiB RAM

        
💰 Cost Tip
Cloud providers charge based on requests, not limits. Keep requests close to actual usage and limits slightly above. Over-provisioning = wasted money.

📈 Step 8: Autoscaling (HPA)


apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: myapp-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: myapp
  minReplicas: 2
  maxReplicas: 10
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70


      
🚪 Step 9: Ingress


apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: myapp-ingress
spec:
  ingressClassName: nginx
  rules:
    - host: myapp.example.com
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: myapp-service
                port:
                  number: 80
